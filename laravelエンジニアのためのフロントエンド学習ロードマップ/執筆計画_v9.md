## 🎯 執筆で気をつけるポイント（Overview）

### 1. 分量の確保【v10で大幅強化】

- **1セクションあたり最低4,000文字**として、重たい学習内容の場合は詳細な説明が必要なので、上限の指定はせずに初学者でも理解しやすい教材にすることを前提にする
- プログラミング未経験・ITリテラシーが高くない学習者を想定し、説明の丁寧さ・わかりやすさを最優先にする
- 内容が薄くならないよう、具体例や実践演習・初学者でもわかりやすい概念に対する丁寧な説明とコードリーディングを行うこと

### 2. 構成の統一

各セクションは以下の構成で統一:

- **🎯 このセクションで学ぶこと:** 学習目標を**行動目標**として明示（「〜ができるようになる」「〜を説明できる」という形式で2〜3項目）
- **🖼️ 完成イメージ:** 何を作るのかがわかる完成イメージ（スクリーンショット、図解など）を提示
- **導入:** なぜこの技術が必要か、実務での具体例
- **概念の説明:** 比喩を交えた分かりやすい解説
- **詳細解説:** コマンド/コードの使い方、オプション、注意点
- **💡 TIP:** 補足情報、よくある間違い、ベストプラクティス
- **🏃 実践:** ステップバイステップのハンズオン
- **コードリーディング:** 実行結果の丁寧な解説
- **✨ まとめ:** 要点を3〜5項目で整理
- **📝 学習のポイント:** チェックリスト形式（3〜5項目）

### 3. 比喩の使い方

- 過度に凝った比喩は避ける（例: 「魔法の杖」「秘密の扉」など）
- 比喩は「日常生活」から選ぶ: 「書類」「ノート」「コピー機」など、誰もが知っているものに例える
- 段階的に理解を深められるよう、前提知識を確認する: 「前のセクションで学んだ〇〇を使います」と明記

### 4. 初学者への配慮

- 専門用語は必ず説明してから使う
- 「簡単です」「当然ですが」などの表現は避ける
- つまずきやすいポイントを先回りして解説
- 「間違えても大丈夫」「何度でもやり直せる」ことを伝え、心理的負担を減らす

### 5. コードリーディングの徹底

- すべてのコード例には、処理の流れを1行ずつ解説
- 実行結果を明示し、何が起きているかを説明
- エラーが出る場合は、その原因と対処法も記載

### 6. 実践重視

- 各セクションに必ず「実践」パートを設ける
- ステップバイステップで手を動かせる内容にする
- 実践後に「何ができるようになったか」を確認

### 7. 表とコードブロックの活用

- 複雑な情報は表形式で整理
- コマンド/コードは必ずコードブロックで記述
- 実行結果もコードブロックで明示

### 8. 絵文字の効果的な活用

- 🎯: 学習目標、次のステップ
- ✨: まとめ
- 💡: TIP、補足情報
- ⚠️: 注意事項
- 🏃: 実践
- 📝: 学習のポイント
- 🔑: 重要な概念
- ⚙️: ツール、コマンド
- 🖼️: 完成イメージ

### 9. 重複の削減

- 同じ概念を別章で繰り返さない
- 前のセクションで説明した内容は、簡潔に参照する
- できる限り演習は統合し、一気通貫で学べるようにする（演習量を確保するための問題等はその限りではない）

### 10. 完成イメージの提示【v9で追加】

- **各チュートリアル・チャプターの冒頭**で、「このチュートリアル/チャプターで何を作るのか」がわかる**完成イメージ**を提示する
- **ハンズオンの冒頭**でも、具体的な完成イメージを提示し、ゴールを明確にする
- 完成イメージは、スクリーンショット、図解、コードの実行結果など、視覚的にわかりやすい形式で提示する

## 🚀 現場直結スキル（8つの重点項目）【FE特化版】

### 1. デバッグ力（Chrome DevToolsの徹底活用）

- **Console, Network, Elements, Sourcesタブの使い分け**を実践的に学ぶ
- エラーメッセージの読み方、スタックトレースの追い方を徹底解説
- **Reactコンポーネントのデバッグ**（React DevToolsの活用）

### 2. 早期リターンとガード節

- **ネストを減らす書き方**を徹底（if文の深さを減らす）
- Reactコンポーネントでの早期リターンの活用

### 3. エラーハンドリングの徹底

- **try-catch**だけでなく、**Error Boundary**や**Suspense**の活用
- **非同期処理のエラーハンドリング**（Loading/Error/Empty状態の管理）

### 4. ディレクトリ構成とファイル命名

- **Feature-based**と**Layer-based**の使い分け
- コンポーネント、hooks、utilsの適切な配置

### 5. 命名力（変数名・関数名・コンポーネント名）

- **意図が伝わる命名**を徹底
- **boolean変数は`is`, `has`, `should`で始める**
- **イベントハンドラは`handle〜`で統一**

### 6. コンポーネント設計（責務の分離）

- **Presentational Component**と**Container Component**の分離
- **単一責任の原則**をコンポーネントに適用

### 7. イミュータビリティ（不変性）の徹底

- **`map`, `filter`, `reduce`を使った配列操作**
- **スプレッド構文**を使ったオブジェクトのコピー
- **直接変更（mutation）を避ける**

### 8. 環境変数の管理

- **`.env`ファイルの適切な管理**
- **Next.jsでの環境変数の扱い**（`NEXT_PUBLIC_`プレフィックス）

## 🔑 重要な設計思想【FE特化版】

### 1. 「状態」と「UI」を分離する

- **状態管理**（useState, useReducer, Zustand, TanStack Query）と**UI表示**を明確に分ける
- **ビジネスロジック**をカスタムフックに切り出す

### 2. 「書く」より「使いこなす」

- **車輪の再発明をしない**（既存のライブラリを活用する）
- **React Hook Form, Zod, TanStack Query**などのデファクトスタンダードを使いこなす

### 3. 「動くコード」より「読めるコード」

- **他者が見て分かりやすいコード**を書く
- **コメントに頼らず、コード自体が説明になる**ように書く

### 4. 「完璧」より「イテレーション」

- **最初から完璧を目指さない**
- **小さく作って、動かして、改善する**

### 5. 「型」で守る

- **TypeScriptの型システム**を活用し、実行時エラーを防ぐ
- **`any`を避け、`unknown`から型を絞る**

### 6. 「テスト」で保証する

- **ユニットテスト**（Vitest）と**E2Eテスト**（Playwright）で品質を保証
- **テストは仕様書**という意識を持つ

### 7. 「パフォーマンス」を意識する

- **不要な再レンダリングを避ける**（React.memo, useMemo, useCallback）
- **Lighthouseで計測**し、改善する

### 8. 「アクセシビリティ」を忘れない

- **セマンティックHTML**を使う
- **キーボード操作**に対応する

### 9. 「チーム」で開発する

- **Storybookでコンポーネントカタログを作る**
- **視覚的なフィードバック**を活用したコードレビュー

### 10. 「ユーザー体験」を最優先する

- **Loading状態**を適切に表示する
- **エラーメッセージ**をわかりやすく表示する

## ✍️ 全チュートリアル共通: ハンズオンセクションの執筆方針

全チュートリアル（Tutorial 1〜12）のハンズオンを含むセクションには、以下の4つの要素を必ず含めます。

### 1. Step形式への統一

| 項目 | 内容 |
|:---|:---|
| 形式 | 全セクションで「Step 1」「Step 2」...の形式で手順を明確化する |
| タイトル | 各Stepには明確なタイトルを付ける（例: 「Step 1: プロジェクトの作成」） |
| サブステップ | Stepの中でさらに細かい手順がある場合は「1-1」「1-2」のようにサブステップを使用する |
| 目的 | 学習者が「今どこにいるか」を常に把握できるようにする |

### 2. 先輩エンジニアの思考プロセス

| 項目 | 内容 |
|:---|:---|
| 配置 | 各セクション冒頭に「🧠 先輩エンジニアの思考プロセス」セクションを追加する |
| 問いかけ | 「なぜこの順番で実装するのか」という問いかけから始める |
| 理由 | 理由を3つ程度挙げて、実装順番の意図を明確にする |
| 表形式 | 「このセクションでやること」を表形式で整理する |
| まとめ | 💡 ポイントで重要な考え方を一言でまとめる |

### 3. コードリーディングの充実

| 項目 | 内容 |
|:---|:---|
| 配置 | すべてのコードブロックの後に詳細な解説を追加する |
| 見出し | 「コードリーディング」という見出しを付けて、1行ずつ解説する |
| 矢印記法 | 「→」を使って各行の意味を簡潔に説明する |
| 初学者目線 | つまずきやすいポイントを先回りして解説する |

### 4. 表形式の活用

| 項目 | 内容 |
|:---|:---|
| 目的 | 情報を整理し、学習者の理解を助けるために表を積極的に使用する |
| 用途 | 比較（メリット/デメリット）、一覧（コマンド/オプション）、対応関係（入力/出力）などに表を使う |
| 効果 | 長い文章よりも表で整理することで、学習者の理解を助ける |
| 補足 | 表の後には必ず補足説明を加える |

### 補足: ハンズオンセクション（*_hands_on.md）の構成（v12で追加）

各チャプター末尾のハンズオンセクションには、以下の構成で「🏃 実践: 一緒に作ってみましょう！」セクションを追加します。

| 項目 | 内容 |
|:---|:---|

## 📋 執筆時の注意事項

### 1. 文字数管理

- **1セクションあたり最低4,000文字**を目標にする
- 内容が薄くならないよう、具体例や実践演習を豊富に含める

### 2. 初学者への配慮

- プログラミング未経験・ITリテラシーが高くない学習者を想定
- 専門用語は必ず説明してから使う
- つまずきやすいポイントを先回りして解説

### 3. 実践性

- 各セクションに必ず「実践」パートを設ける
- 実務での具体例を豊富に含める

### 4. 一貫性

- 全セクションで構成を統一する
- 絵文字の使い方を統一する

### 5. 参考資料の引用

- 公式ドキュメントへのリンクを適切に含める
- 最新情報を反映する
